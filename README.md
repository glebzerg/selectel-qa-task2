## ✅ Что тестируется

Проект покрывает **4 области** тестирования:

### 1. 🏠 Главная страница (`test_main_page.py`)
| Проверка | Тип | Зачем |
|---|---|---|
| HTTP 200 на главной | API | Сайт доступен — клиенты могут зайти |
| Редирект HTTP → HTTPS | API | Безопасность и SEO |
| Время ответа сервера < 5с | API | Задержка > 3с увеличивает bounce rate на 32% |
| Заголовки безопасности | API | Защита от XSS, clickjacking |
| Наличие H1 (ровно один) | UI | SEO и доступность |
| Видимость header и footer | UI | Структурная целостность страницы |
| Отсутствие горизонтального скролла | UI | Качество вёрстки |
| Meta description (50–320 символов) | UI | CTR из поисковой выдачи |
| Open Graph теги (title, image) | UI | Корректный шаринг в соцсетях |
| Canonical URL | UI | Нет дублирования контента |
| Favicon | UI | Идентичность бренда во вкладке |
| `<html lang="ru">` | UI | Скринридеры и голосовые помощники |
| Alt у всех изображений | UI | WCAG 2.1 (доступность) |
| Теги `<main>` и `<nav>` | UI | Семантика и доступность |

### 2. 🧭 Навигация и роутинг (`test_navigation.py`)
| Проверка | Тип | Зачем |
|---|---|---|
| HTTP-статус 9 ключевых страниц | API | Критичные разделы должны быть доступны |
| Несуществующий URL → 404 | API | Поисковики не индексируют мусор |
| 404-страница с полезным контентом | UI | Удержание пользователя вместо тупика |
| www → non-www редирект | API | Нет дублирования контента |
| Слеш в конце URL — консистентность | API | Canonical-проблемы |
| Внешние ссылки в новой вкладке | UI | UX — не уводить пользователя с сайта |
| robots.txt доступен | API | Управление индексацией |
| sitemap.xml доступен | API | Поисковики находят все страницы |

### 3. 📱 Мобильная версия (`test_mobile.py`)
| Проверка | Тип | Зачем |
|---|---|---|
| Нет горизонтального скролла на 390px | UI | Мобильный трафик — 50–70% пользователей |
| H1 виден без скролла | UI | Пользователь сразу видит суть страницы |
| Viewport meta-тег присутствует | UI | Без него мобильники рендерят как десктоп |
| Рендер на 4 разрешениях | UI | iPhone, Android, iPad (портрет и альбом) |
| Touch-цели ≥ 44×44px | UI | WCAG 2.5.5 — удобство нажатия пальцем |

### 4. 📄 Контент и UX (`test_content.py`)
| Проверка | Тип | Зачем |
|---|---|---|
| Ссылка на регистрацию есть на главной | UI | Основной CTA для новых клиентов |
| Ссылка на вход в кабинет есть | UI | Существующие клиенты не звонят в поддержку |
| Страница регистрации доступна | API | my.selectel.ru/registration возвращает 200 |
| Страница цен содержит цены | UI | Клиент принимает решение на основе цен |
| Нет Lorem ipsum / placeholder | UI | Контент готов к продакшену |
| Нет битых картинок | UI | Сломанные картинки = плохой имидж |
| Нет критичных JS-ошибок | UI | Интерактивные элементы работают |

**Итого: 35+ автоматизированных проверок**

## 🛠 Стек инструментов

| Инструмент | Роль | Почему выбран |
|---|---|---|
| **Python 3.12** | Язык | Стандарт в QA-автоматизации, читаемый синтаксис |
| **Playwright** | Браузерная автоматизация | Надёжнее Selenium, встроенные ожидания, поддержка мобильных |
| **pytest** | Тест-фреймворк | Гибкие фикстуры, параметризация, маркеры |
| **pytest-playwright** | Интеграция | Playwright-фикстуры из коробки |
| **pytest-html** | Отчёты | HTML-отчёт без внешних сервисов |
| **requests** | HTTP-проверки | Быстрые API-тесты без запуска браузера |
| **Page Object Model** | Паттерн | Устойчивость к изменениям вёрстки |

## 📁 Структура проекта

```
selectel-qa-tests/
│
├── tests/                      # Все тест-файлы
│   ├── test_main_page.py       # Главная страница: HTTP, SEO, структура, a11y
│   ├── test_navigation.py      # Роутинг, 404, редиректы, robots.txt
│   ├── test_mobile.py          # Адаптивность, viewport, touch-цели
│   └── test_content.py         # UX, контент, JS-ошибки, CTA
│
├── pages/                      # Page Object Model
│   ├── base_page.py            # Базовый класс: навигация, утилиты
│   ├── main_page.py            # Локаторы и действия главной страницы
│   └── navigation_page.py     # Локаторы и действия для проверки навигации
│
├── utils/
│   └── helpers.py              # Вспомогательные функции и константы
│
├── reports/                    # HTML-отчёты (генерируются при запуске)
│
├── .github/
│   └── workflows/
│       └── tests.yml           # CI/CD через GitHub Actions
│
├── conftest.py                 # Глобальные pytest-фикстуры (браузер, контексты)
├── pytest.ini                  # Настройки pytest и маркеры
├── requirements.txt            # Зависимости проекта
├── .env.example                # Шаблон переменных окружения
└── README.md                   # Этот файл
```

**Почему такая структура?**
- **Page Object Model** — при изменении вёрстки правится только файл в `pages/`, а не каждый тест
- **conftest.py** — фикстуры браузера создаются один раз и переиспользуются
- **utils/helpers.py** — константы в одном месте, легко изменить BASE_URL для другого окружения

## 🚀 Быстрый старт

### Предварительные требования

- Python 3.10 или выше
- pip
- Git

### Установка

```bash
# 1. Клонировать репозиторий
git clone https://github.com/YOUR_USERNAME/selectel-qa-tests.git
cd selectel-qa-tests

# 2. Создать виртуальное окружение (рекомендуется)
python -m venv venv

# Активация на macOS/Linux:
source venv/bin/activate

# Активация на Windows:
venv\Scripts\activate

# 3. Установить зависимости
pip install -r requirements.txt

# 4. Установить браузеры Playwright
playwright install chromium

# 5. Скопировать конфиг окружения (опционально)
cp .env.example .env
```

### Проверка установки

```bash
# Должен вывести версию pytest
pytest --version

# Должен вывести список всех тестов без запуска
pytest --collect-only
```

## ▶️ Запуск тестов

### Запустить все тесты

```bash
pytest
```

### Запустить только smoke-тесты (быстрая проверка, ~1–2 мин)

```bash
pytest -m smoke
```

### Запустить конкретный файл

```bash
pytest tests/test_main_page.py
```

### Запустить конкретный тест по имени

```bash
pytest -k "test_main_page_returns_200"
```

### Только API-тесты (без браузера, быстро)

```bash
pytest -m api
```

### Только UI-тесты (с браузером)

```bash
pytest -m ui
```

### Только мобильные тесты

```bash
pytest -m mobile
```

### Параллельный запуск (быстрее в 2–4 раза)

```bash
pytest -n auto
```

### С подробным выводом

```bash
pytest -v
```

### С остановкой на первой ошибке

```bash
pytest -x
```

### Полный регрессионный прогон

```bash
pytest -m regression -v --html=reports/regression.html
```

## 📊 Просмотр отчётов

После запуска тестов HTML-отчёт создаётся автоматически:

```bash
# Открыть отчёт в браузере (macOS)
open reports/report.html

# Linux
xdg-open reports/report.html

# Windows
start reports/report.html
```

Отчёт содержит:
- ✅ Список пройденных тестов с временем выполнения
- ❌ Упавшие тесты с полным traceback и причиной
- ⚠️ Пропущенные тесты с объяснением
- 📊 Сводную статистику по тест-сессии

## 🏷 Маркеры тестов

| Маркер | Описание | Когда запускать |
|---|---|---|
| `smoke` | Минимальный набор, проверяет что сайт «жив» | При каждом деплое (< 2 мин) |
| `regression` | Полная проверка всей функциональности | Раз в день или перед релизом |
| `ui` | Тесты с запуском браузера через Playwright | При изменениях вёрстки |
| `api` | HTTP-тесты без браузера | Быстрая проверка доступности |
| `mobile` | Тесты мобильного отображения | При изменениях CSS/адаптивности |
| `accessibility` | Проверки доступности (WCAG) | При обновлении контента/вёрстки |
| `negative` | Негативные сценарии (ошибки, 404) | В регрессионном прогоне |

## ⚙️ CI/CD

Проект настроен для автоматического запуска через **GitHub Actions**.

Тесты запускаются:
- При каждом `push` в ветку `main` или `develop`
- При открытии Pull Request в `main`
- Каждый день в 12:00 МСК (расписание cron)
- Вручную через кнопку «Run workflow» в GitHub UI

Файл конфигурации: `.github/workflows/tests.yml`

После прогона HTML-отчёт сохраняется как **GitHub Actions Artifact** (хранится 14 дней).  
Скачать: вкладка **Actions → выбрать запуск → Artifacts**.

## 🏗 Архитектурные решения

### Page Object Model (POM)
Все локаторы и действия хранятся в классах в папке `pages/`.  
Тесты не содержат CSS-селекторов напрямую.

**Преимущество**: изменилась вёрстка кнопки → правим только `main_page.py`, а не 20 тест-файлов.

### Разделение API и UI тестов
HTTP-проверки (доступность, статус) выполняются через `requests` — без браузера.  
Это быстрее в 5–10 раз и не зависит от JavaScript.

### Изолированные контексты браузера
Каждый тест получает **чистый** контекст браузера (куки, localStorage обнулены).  
Тесты не влияют друг на друга.

### Параметризация
Одна функция `test_important_page_is_accessible` проверяет 9 страниц через  
`@pytest.mark.parametrize` — нет дублирования кода.
